JavaScript Testing w/ Jasmine

You can now write JavaScript tests within the codebase using Jasmine! 

It is suggested that you do not test the UI/DOM with Jasmine, that is more the responsibilty of the autmoated UI tests (selenium). Treat them more like unit tests for C# code.

Be warned the build server is currently setup to fail the build if any JavaScript tests are failing.

Creating Tests

Writing tests

  1. Create a ".spec.js" file next to the file you are testing. e.g. If you are testing the file MyFile.js then the test file would be called MyFile.spec.js.
  2. Set the build action of the new specification file to "None". We don't want the file going to release, it is only for testing purposes.
  3. Write your specification in the specification file.

You define a specification using the describe function and test aspects of it with the it function as shown below. 
It is meant to be in human readable language to that when the specifcations are run it would read: "Claculator can add two numbers - Success!".
You can find out more about other features such as spys, mocks, teardown functions etc. at https://jasmine.github.io/tutorials/your_first_suite



Adding your tests to the "specification runner"

Once you have writting your specification file you will need to add it to the specification runner. This is the file that evaluates all of the tests and is used by the build server.

Within the file "jasmine-spec-runner.html" add your specifcation files and source files as detailed below.


Running Tests

To run the tests and see the  their results you can simply open the "jasmine-spec-runner.html" within a browser of your choosing. It will then either show a green page if all the specifications passed or a red page stating which tests have failed (as shown below).




Debugging Tests

Option A - Browser Development tools

To debug a test you can use a browsers developer tools, just like debugging a website. 

  • Open the "jasmine-spec-runner.html" file in your browser of choice
  • Open up Dev Tools
  • Select the "Sources Tab"
  • Select the .spec.js file you want to debug
  • Put a break point in the exact specification(s) you want to tests
  • Refresh the page (to get it to re-evaluate the tests and trigger the break poitns).

Option B - Visual Studio Extensions

Along with several plugins available there for VisualStudio there is one called Chutzpah Test Runer Context Menu Extension (https://github.com/mmanela/chutzpah) which will allow you to right click on a test and execute it in run or debug mode. 

  • Run JS Tests - Will output the results to VisualStudio's console
  • Open in browser - Will open an autogenerated spec-runner html file showing the results in the browser
  • Run Chutzpah With -> Debugger - Will allow you to put breakpoints within the code to debug your tests



Integrating with Team City

There is a individual build step for the JavaScript testing which runs a custom script. It uses phantom.js to execute the tests with two parameters (phantom.js has to exist on the server):

  1. jasmine-teamcity.js - The file used for evaluating the jasmine test results and putting it into a format that TeamCity can understand.
  2. jasmine-spec-runner.html - The file where all the tests are located (as script tags)

Run:

Custom Script

Command Script:

cd d:/phantom
phantomjs.exe %system.teamcity.build.workingDir%/Source/Web/UI/JavaScriptTests/Jasmine/jasmine-teamcity.js file:///%system.teamcity.build.workingDir%/Source/Web/UI/JavaScriptTests/Jasmine/jasmine-spec-runner.html
